The AHA! User Model

For every individual user AHA! maintains a user model, consisting of concepts with attributes. As an author you never deal with the user model directly, but influence the possible updates to the model through the concept structure and the associated event-condition-action rules. Whether the user model is stored in the form of XML files or in the mySQL database, it contains the same information:

  • There is a pseudo concept called personal which represents information about the user, independent of the application domain. It can have arbitrarily many attributes that are created through an application's registration form. If the registration form contains a form field: 
    < input type="hidden" name="myattribute" value="mystring" />
    then an attribute myattribute will be added to the concept personal and it will get the value mystring. Of course you can use any other type of field (e.g. text instead of hidden) to let the user enter information in the field.
    The concept personal can be used like any other concept. Attribute values can be used in event-condition-action rules to create behavior that depends on the information a user entered in the registration form. But apart from author-chosen attributes that have no other meaning than their use in rules there are some predefined attributes of the personal concept that have a meaning for AHA!:
    • name: This is the end-user's name, entered in the registration form. The name can be included in a header or footer, using the <username/> variable. Anonymous users do not have a name, although a standard name can be created through the registration form.
    • email: This is the end-user's email address, entered in the registration form. The address can be included in a header or footer using the <email/> variable. Anonymous users do not have an email address.
    • id: This the login id chosen by the end-user or generated automatically for an anonymous user. The login id can be included in a header using the <loginid> variable.
    • password: This is the end-user's password. In AHA! 2.0 the password is not stored in encrypted form. As a result access to the directory with the user models (in XML format) or to the mySQL database should be restricted. Anonymous users cannot have a password.
    • university: This is one of the attributes that date back to the original application area: on-line courses offered to several universities. Of course it can be used to represent something else, like the end-user's company. This attribute can be included in a header using the <university> variable.
    • department: This is the second attribute for courses. It can represent the end-user's department, course major, or anything else. This attribute can be included in a header using the <department> variable.
    • course: Each AHA! application is identified as a "course" (for historical reasons). This attribute is also the name of the subdirectory of an AHA! server containing the application. This attribute can be included in a header using the <course> variable.
    • title: Each AHA! application can be identified using a "title". This attribute can be included in a header using the <coursename> variable. (Not "title" for implementation reasons.) In this tutorial the header shows "knowledge of AHA! tutorial". The part "AHA! tutorial" is the title of this application.
    • goodlink: This is the color used for links to suitable pages that have not been visited. It is a string value representing the RGB value of the color as a hexadecimal number.
    • neutrallink: This is the color used for links to previously visited suitable pages.
    • badlink: This is the color of links to unsuitable pages.
    • activelink: This is the color of links while the mouse button is pressed to follow the link.
    • externallink: This is the color of links to external pages that have not been visited.
    • externalvisitedlink: This is the color of links to visited external pages.
    • background: This is the name for the background image used on every page. (In the tutorial this is "/aha/AHAtutorialLS/icons/tuebg.gif".)
  • All other concepts belong to the application domain. The concept structure determines which concepts exist and which attributes they have. For every attribute of a concept the user model stores: the type of the attribute (Boolean, integer or string), the value, and a Boolean to indicate whether the attribute value is persistent.
