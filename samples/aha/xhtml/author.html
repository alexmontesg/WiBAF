Authoring AHA! Applications

Authors in AHA! are special users, created by the manager (who installs AHA!). As an author you can access the authoring tools and your account information through the "Author Workplace".

Creating AHA! applications involves four main steps: writing pages, creating a conceptual structure of the aplication, defining the adaptation rules, and defining the look and feel or layout of the application.

  • For each application you must create a directory tree of which the root (like "AHAtutorialLS" for this tutorial) we will refer to as the "course" (for historical reasons). In the root directory you must create an "index.html" file with a login form and a link to another file to register as a new user. In this tutorial this file is "Registration.html". The forms must adhere to the login form syntax which is fairly obvious from looking at the tutorial version. From the forms the servlets "Login", "Register" or "Anonymous" are called. Some form elements create variables that can be retrieved and used in a header or footer. Additional elements can be added. They will create attributes for the personal pseudo-concept in the user model.
  • AHA! version 2.0 requires pages to be written in XHTML, the standard markup language defined by the World Wide Web Consortium. We have a brief explanation of the differences between the "old" plain html and xhtml. AHA! 2.0 added some tags to this language for the generation of optional headers and footers and for the conditional inclusion of fragments. In AHA! 3.0 the standard XHTML "object" tag can be used to achieve the same functionality (and more) without using an AHA!-extended document format.
  • Each application deals with concepts. In the simplest case each page is a concept and vice versa. But often you wish to consider "higher level" concepts. AHA! comes with a "high level" graph author tool to define the structure of concepts and pages. You can define relationships of different types between concepts. The tool generates the event-condition-action rules from this graph.
  • The adaptation in AHA! is a two-stage process. As an author you define rules that translate user interactions (page accesses) into user model updates. The user model instance is then used to determine how the presentation is adapted. There are three ways to author the adaptation rules: you can generate rules from concept relationships using the graph author, you can manually create rules using the concept editor, or you can manually edit an XML file containing the concepts and adaptation rules.
  • An additional part of AHA! consists of forms for letting end-users change their user model. There is a Form Editor through which you can define such forms and thus control which parts of the user model an end-user can update and which part remains under strict control of the AHA! system. For updates to color preferences and knowledge about concepts there are standard forms that can be accessed through the header or footer.
  • For educational applications AHA! also offers the possibility to present and evaluate multiple-choice tests. The tests can be randomized to avoid circulation of lists of answers among students.

The authoring tools all interact with XML files stored in a special directory for "author files". This directory, called "graphauthorfiles" in the AHA! configuration file is predefined as "/author/authorfiles". When you make changes to these files (with or without AHA! tools) you need to convert the XML files to AHA!'s internal format, as described on the manager page.
Note that for the Form Editor you need to (manually) copy or move the created files to the directory containing the pages of your application.

Warning: the graph author generates a file that can be read by the concept editor, but it does not read these files. Changes made to the concept/rule structure in the concept editor are lost when you later make changes using the graph author.
